# 09 — Execution guide for agent-driven implementation

Depends on: [00-epic](00-epic.md). Read this before executing any ticket in
this epic.

> **Enforced form:** the session protocol below is gatified as the repo skill
> `.claude/skills/fix-ticket/` — invoke `/fix-ticket` with the ticket path and
> the sequence is enforced by `steps/gate.sh` rather than read-and-remember.
> This prose remains the rationale; the skill is the pipeline. It exists because capable models fail these tickets in predictable
ways; every rule below maps to a known failure mode.

## Session protocol

- **One ticket per session, per branch.** Fresh context beats a long session
  carrying seven tickets of stale state. Order: 05 before 06 (mandatory,
  see [06-demo-toggle.md](06-demo-toggle.md)); everything else independent.
- **Re-verify every `file:line` before editing.** Citations in these tickets
  were accurate against `feat/popup-redesign-landing` at review time; earlier
  tickets landing will shift them. `grep -n` for the quoted code, never trust
  a line number you haven't reproduced this session.
- **Test-first, both outputs pasted.** Write the disconfirming test, run it,
  paste the FAILING output. Implement. Run again, paste the PASSING output.
  A criterion with only the passing half is unmet (root `CLAUDE.md`,
  Acceptance criteria section).
- **Never modify an existing assertion to make a suite pass.** If an assertion
  looks wrong, stop and report it as a finding instead. Same for skipping,
  loosening timeouts, or `test.fixme`.
- **Diff scope = ticket scope.** No opportunistic refactors, renames, or
  formatting in files the ticket doesn't name. If you find an adjacent bug,
  report it; don't fix it in the same diff.
- **No commits or pushes unless Ian asks in the moment.** Leave the working
  tree ready; never commit local test recordings.
- **Verification is command output only.** There is no manual browser to look
  at; "verified visually" is a hallucination by construction. Drive everything
  through the harness (`test/e2e/harness.mjs`), `npm test`, or
  `node scripts/screenshots.mjs` and paste what they printed.

## Known traps in this codebase

- **Two copies of the component until 05 lands.** Editing `openjam-popup.js`
  does NOT change the landing page: `docs/index.html:531` holds a full pasted
  copy. After 05, the reverse rule: the block between the splice markers in
  `docs/index.html` is generated by `node build.mjs`; edit `openjam-popup.js`
  and rebuild, never the block.
- **Playwright visibility lies for collapsed shadow content.** An element
  inside `.mic-body` at `max-height:0; opacity:0` (`openjam-popup.js:113`)
  still passes `.waitFor()`/visibility checks (this exact trap shipped as
  `scripts/screenshots.mjs:46`, see [08](08-test-integrity.md)). Assert
  bounding-box height or `option` counts, not element presence.
- **Locators pierce open shadow roots
  (https://playwright.dev/docs/locators#locate-in-shadow-dom); `querySelector`
  doesn't.** Inside
  `evaluate()` you must hop via `el.shadowRoot.querySelector(...)` — a bare
  `document.querySelector('.err')` silently returns null and can make a
  wrongly-written disconfirming test "fail" for the wrong reason. Confirm a
  disconfirming test fails on the assertion you intended, not on a selector error.
- **`background.js` sets `session.recording = true` early**
  (`background.js:374`), before debugger attach completes. Any test reasoning
  about start/stop timing must account for that window (it's the root of
  [01](01-toggle-reentrancy.md)).
- **Build before judging the landing page.** `npm test` runs `npm run build`
  first (root `CLAUDE.md`, Tests & CI); a stale `docs/index.html` after 05
  means you're inspecting last build's output.
- **The extension is live on the Chrome Web Store.** `manifest.json` version
  bumps and release tags (`v*` auto-release) are out of scope for every ticket
  in this epic.

## Done means

`npm test` green, every acceptance criterion in the ticket satisfied with
pasted command output including its disconfirming input, diff contains only
the ticket's scope, and open questions reported rather than resolved by
invention. "Couldn't verify X" is an acceptable and reportable outcome
(root `CLAUDE.md`); a confident claim without output is not.
